## Notes

### Coverage summary

- **18 acceptance criteria total** (AC1.1 through AC4.3)
- **18 have automated test coverage** in at least one form (unit test, integration test, benchmark, or script-based E2E check)
- **9 additionally require human verification** because they involve hardware-dependent throughput measurements, Pareto dominance interpretation, or statistical thresholds over corpora

### Test execution tiers

1. **Fast (< 30s):** Unit tests in `src/lz77.rs`, `src/lzseq.rs`, `src/optimal.rs`, `src/pipeline/mod.rs` -- run via `cargo test`. Covers AC1.1, AC1.2, AC1.4, AC1.6, AC2.4, AC3.5, AC3.6, AC4.3 (heuristic logic).
2. **Medium (1-5 min):** Integration tests in `src/pipeline/tests.rs`, `src/pipeline/parallel.rs`, `src/webgpu/tests.rs` -- run via `cargo test --features webgpu`. Covers AC3.2, AC3.3, AC3.4, AC3.5, AC3.6.
3. **Slow (5-20 min):** Benchmark scripts (`scripts/bench.sh`, `scripts/pareto-bench.sh`, `scripts/analyze-ratio.sh`) -- run manually or in CI. Covers AC1.3, AC1.5, AC2.1, AC2.2, AC2.3, AC3.1, AC4.1, AC4.2.
4. **Manual (20+ min):** Full corpus runs (`scripts/tune-selector.sh`, `scripts/pareto-bench.sh --corpus both`), GPU benchmarks (`cargo bench --features webgpu`). Covers AC4.3 (full validation), AC3.2, AC3.3, AC3.4 (throughput comparison).

### GPU-dependent tests

Tests for AC3.2, AC3.3, AC3.4, and AC3.6 require the `webgpu` feature and a GPU device. All GPU tests are gated with `#[cfg(feature = "webgpu")]` and skip gracefully (return early, not fail) when no GPU is detected. CPU-only correctness is verified separately via `cargo test --no-default-features`.

### Decompression coverage (AC2.3)

AC2.3 (decompression throughput) is primarily a Phase 4 concern (rANS SIMD decode wiring) but is validated in Phase 8. If Phase 4's SIMD improvement is insufficient, the Phase 4 plan (Task 3) includes a remediation note: investigate branchless decode loops, prefetching, and lookup table optimization. The decompression path is tested for correctness throughout all phases via round-trip tests, but throughput requires dedicated benchmarking.

### Known pre-existing gaps

As of the codebase snapshot (2026-02-22), LzSeqR ratio on Canterbury is measured at 32.0% vs gzip-6 at 28.6% -- a 3.4 percentage point gap. The Phase 2 (match quality) and Phase 3 (optimal parsing) implementations target closing this gap. If the gap persists after both phases, Phase 8 Task 9 documents it as a known delta rather than a blocker.
